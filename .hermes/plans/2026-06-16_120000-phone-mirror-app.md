# Phone Mirror App — Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Build a cross-platform desktop+web app that mirrors an Android/iOS phone screen to a PC or Mac via USB-C, Wi-Fi, or Bluetooth — with screen recording, PC input control, and low-latency streaming.

**Architecture:** Rust core library for protocol handling and video pipeline, wrapped in a Tauri desktop app with a React/TypeScript web frontend. Android support first via ADB/scrcpy protocol, iOS-ready architecture with SCRP integration as Phase 2.

**Tech Stack:**
- Rust (core library — protocols, video, input)
- Tauri v2 (desktop shell, cross-platform)
- React + TypeScript (web UI)
- ffmpeg (video encoding/decoding)
- ADB protocol (Android screen mirroring via `framebuffer` streaming)
- SCRP/libimobiledevice (iOS, Phase 2)
- vlc or gstreamer for playback pipeline (or custom via rav1e/avif)

```rust
// src-tauri/src/adb/device.rs
use serde::{Deserialize, Serialize};
use std::process::Command;

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Device {
    pub serial: String,
    pub state: String,
    pub model: String,
}

pub fn list_devices() -> Result<Vec<Device>, String> {
    let output = Command::new("adb")
        .args(["devices", "-l"])
        .output()
        .map_err(|e| format!("ADB command failed: {}", e))?;

    let stdout = String::from_utf8_lossy(&output.stdout);
    let mut devices = Vec::new();

    for line in stdout.lines().skip(1) {
        let parts: Vec<&str> = line.split_whitespace().collect();
        if parts.len() >= 2 && parts[1] == "device" {
            let serial = parts[0].to_string();
            let model = parts.iter()
                .skip(2)
                .find(|p| p.starts_with("model:"))
                .map(|p| p.strip_prefix("model:").unwrap_or("").to_string())
                .unwrap_or_else(|| "Unknown".to_string());

            devices.push(Device {
                serial,
                state: "device".to_string(),
                model,
            });
        }
    }

    Ok(devices)
}

pub fn connect_wifi(device: &str) -> Result<(), String> {
    let output = Command::new("adb")
        .args(["connect", device])
        .output()
        .map_err(|e| format!("ADB connect failed: {}", e))?;

    let stdout = String::from_utf8_lossy(&output.stdout);
    if stdout.contains("connected") {
        Ok(())
    } else {
        Err(format!("Failed to connect: {}", stdout))
    }
}
```

**Step 1: Implement screen stream capture**

The approach mirrors scrcpy — use `adb shell screenrecord` for a simple start, then graduate to H.264 streaming via `adb exec-out` with the `--stream` flag.

```rust
// src-tauri/src/video/stream.rs
use std::process::{Child, Command, Stdio};
use std::sync::mpsc;
use tokio::time::{interval, Duration};

pub struct ScreenStream {
    child: Option<Child>,
    serial: String,
}

impl ScreenStream {
    /// Start screen capture via ADB screenrecord (initial approach)
    pub fn start_h264(device_serial: &str) -> Result<Self, String> {
        // Use the scrcpy approach: push scrcpy-server or use screenrecord
        // Phase 1: screenrecord (simpler but no encoding control)
        // Phase 2: scrcpy-server H.264 stream (better quality + lower latency)

        let child = Command::new("adb")
            .args([
                "-s", device_serial,
                "exec-out", "screenrecord", "--profile", "normal", "-",
            ])
            .stdout(Stdio::piped())
            .stderr(Stdio::piped())
            .spawn()
            .map_err(|e| format!("Failed to start screenrecord: {}", e))?;

        Ok(ScreenStream {
            child: Some(child),
            serial: device_serial.to_string(),
        })
    }

    /// Start screen capture via scrcpy server (production approach)
    pub fn start_scrcpy_server(device_serial: &str) -> Result<Self, String> {
        // Push scrcpy-server.jar to device
        let server_path = "/data/local/tmp/scrcpy-server.jar";
        Command::new("adb")
            .args([
                "-s", device_serial,
                "push", "assets/scrcpy-server.jar", server_path,
            ])
            .output()
            .map_err(|e| format!("Failed to push server: {}", e))?;

        // Start server
        let child = Command::new("adb")
            .args([
                "-s", device_serial,
                "shell", "CLASSPATH=", server_path,
                "app_process", "/", "com.genymobile.scrcpy.Server", "2",
            ])
            .stdout(Stdio::piped())
            .stderr(Stdio::piped())
            .spawn()
            .map_err(|e| format!("Failed to start scrcpy server: {}", e))?;

        Ok(ScreenStream {
            child: Some(child),
            serial: device_serial.to_string(),
        })
    }

    pub fn stop(&mut self) -> Result<(), String> {
        if let Some(mut child) = self.child.take() {
            child.kill().map_err(|e| format!("Failed to stop: {}", e))?;
            child.wait().map_err(|e| format!("Failed to wait: {}", e))?;
        }
        Ok(())
    }
}

impl Drop for ScreenStream {
    fn drop(&mut self) {
        let _ = self.stop();
    }
}
```

```tsx
const KEY_MAP: Record<string, string> = {
  Enter: '66',
  Backspace: '67',
  Tab: '61',
  Escape: '111',
  Space: '62',
  // ... more mappings
};

export function KeyboardHandler({ serial }: { serial: string }) {
  useEffect(() => {
    const handler = (e: KeyboardEvent) => {
      const code = KEY_MAP[e.key];
      if (code) {
        e.preventDefault();
        invoke('device_key', { serial, key: code });
      } else if (e.key.length === 1) {
        invoke('device_text', { serial, text: e.key });
      }
    };
    window.addEventListener('keydown', handler);
    return () => window.removeEventListener('keydown', handler);
  }, [serial]);

  return null;
}
```

## Risks & Tradeoffs

1. **H.264 decode latency** — Decoding H.264 in Rust with ffmpeg-next can be tricky. May need to fall back to `ffmpeg` CLI subprocess if the library bindings are insufficient.
2. **iOS SCRP complexity** — iOS screen mirroring requires device pairing (Trust this computer) and may require a signed plist. This is significantly harder than Android. Consider Phase 2.
3. **macOS Tauri dependencies** — libwebkit2gtk is Linux-only; macOS uses WKWebView. Build environment needs proper setup.
4. **Android version compatibility** — ADB screenrecord/scrcpy behavior varies across Android versions. Test on Android 10+ minimum.
5. **Performance** — Real-time H.264 decode + canvas rendering in Tauri may have higher latency than native scrcpy. Consider hardware acceleration.
